## 1. Назначение и статус

Профиль определяет **Salt API**, который оригинал Кельнского цифрового санскритского
словаря (`sanskrit-lexicon.uni-koeln.de`, реализация в `csl-apidev`) ДОЛЖЕН предоставлять,
чтобы клиент, написанный для API C-SALT, обращался к оригиналу без изменений.

Профиль **совместим на уровне протокола** с API C-SALT / Kosh: каждая конечная точка,
параметр, значение перечисления и поле верхнего уровня в точности воспроизводят C-SALT. CSL
добавляет ровно одно: вложенный объект `csl` в записи — данные, которым в модели Kosh нет
места.

Это профиль «чистой комнаты» (?): C-SALT используется только как **эталон для сверки**;
реализация оригинала восстанавливается из данных CSL, а не копируется из C-SALT.

Ключевые слова ДОЛЖЕН, НЕ ДОЛЖЕН, СЛЕДУЕТ, МОЖЕТ — по RFC 2119.

## 2. Конечные точки

Для словаря с идентификатором в нижнем регистре `{id}` (§7) совместимый хост ДОЛЖЕН
предоставлять:

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/dicts/{id}/restful/entries` | Поиск записей (§3, §4). |
| GET | `/dicts/{id}/restful/ids` | Пакетная выборка записей по списку id (§5). |
| POST | `/dicts/{id}/graphql` | GraphQL `entries` и `ids` (§6). |
| GET | `/{dict}/{ref}` | Постоянная «чистая» ссылка на запись по заголовочному слову или lnum — поглощает COLOGNE#249 (§6a). |

Ответы конечных точек `restful` и `graphql` ДОЛЖНЫ иметь тип `application/json`.

## 3. REST — поиск (`/restful/entries`)

### 3.1 Параметры запроса

| Имя | Обязат. | Тип | Значения / перечисление |
|---|---|---|---|
| `field` | да¹ | строка | `id`, `headword_slp1`, `sense`, `re_headwords_slp1`, `created`, `xml` |
| `query` | да | строка | строка поиска |
| `query_type` | да¹ | строка | `term`, `fuzzy`, `match`, `match_phrase`, `prefix`, `wildcard`, `regexp` |
| `size` | нет | целое | максимум записей (по умолчанию 25) |
| `transLit` | нет | строка | `slp1`, `deva`, `hk`, `roman`, `itrans` — **расширение CSL** (по умолчанию `slp1`) |
| `accent` | нет | строка | `yes`, `no` — **расширение CSL** (по умолчанию `no`) |

¹ C-SALT помечает `field`, `query`, `query_type` как обязательные. Хост CSL МОЖЕТ
смягчить `field` и `query_type` до значений по умолчанию `headword_slp1` / `term` для
удобства браузера, но ДОЛЖЕН по-прежнему принимать строгую форму C-SALT. Два **расширения
CSL** НЕ ДОЛЖНЫ менять смысл какого-либо параметра C-SALT.

### 3.2 Ответ

```json
{ "data": { "entries": [ /* запись, §8 */ ] } }
```

При отсутствующем или неверном параметре хост ДОЛЖЕН вернуть HTTP 400 с телом
`{ "error": "<сообщение>" }` (форма сообщения C-SALT:
`Missing or invalid parameter: 'field'`).

### 3.3 Пример

```
GET /dicts/mw/restful/entries?field=headword_slp1&query=agni&query_type=term&size=1
```

## 4. Режимы поиска (`query_type`)

Совместимый хост ДОЛЖЕН принимать все семь значений. *Качество* каждого режима МОЖЕТ
зависеть от поискового бэкенда (см. дорожную карту §4.3: PHP по заголовочным словам или
Elasticsearch). Как минимум:

| Режим | Смысл | Минимальное поведение CSL |
|---|---|---|
| `term` | точное совпадение | точное совпадение ключа-заголовка |
| `prefix` | по началу | совпадение по префиксу заголовка |
| `wildcard` | шаблон `*`/`?` | шаблон по искомому полю |
| `regexp` | регулярное выражение | regex по искомому полю |
| `fuzzy` | по расстоянию редактирования | приближенное совпадение заголовка |
| `match` | токенное совпадение | токенное совпадение по полю |
| `match_phrase` | совпадение фразы | совпадение фразы по полю |

`match` / `match_phrase` по `field=sense` или `field=xml` (поиск по телу статьи) МОЖЕТ быть
отложен (дорожная карта, фаза 4), но тогда ДОЛЖЕН возвращать HTTP 400, а не молча пустой
результат.

## 5. REST — пакетная выборка (`/restful/ids`)

```
GET /dicts/{id}/restful/ids?ids={id}&ids={id}…
```

`ids` — повторяемый (многозначный) параметр. Ответ ДОЛЖЕН быть
`{ "data": { "ids": [ /* запись */ ] } }`. Это **выборка по id**, а не поиск.

## 6. GraphQL (`/graphql`)

Схема — [`salt-api.graphql`](../data/schema/salt-api.graphql). Два корневых поля:

```graphql
entries(field: Field = headword_slp1, query: String!, queryType: QueryType = term, size: Int = 25): [Entry!]
ids(ids: [String!]!): [Entry!]
```

**Регистр.** GraphQL использует camelCase в именах аргументов и полей (`queryType`,
`headwordSlp1`, `reHeadwordsSlp1`); REST — формы snake_case из §3.1 и §8. Обе формы
нормативны; хост ДОЛЖЕН соблюдать регистр вызываемой стороны.

**Имя типа по словарю.** Действующий сервис C-SALT именует тип записи по идентификатору
словаря (`mw`, `ap90`, …). Хост CSL МОЖЕТ следовать этой конвенции; профиль обозначает тип
абстрактно как `Entry`.

## 6a. Постоянные «чистые» ссылки (поглощают COLOGNE#249)

`csl-apidev` уже планирует «чистые» постоянные ссылки (заготовка конечной точки `cleanurl`
в `doc/readme.md`, задача COLOGNE#249), например `/MW/bAQa` (по заголовочному слову) или
`/MW/144239` (по lnum). Профиль Salt **поглощает** эту задачу, а не конкурирует с ней:
совместимый хост CSL СЛЕДУЕТ предоставлять

```
GET /{dict}/{ref}
```

где `{dict}` — код словаря (в верхнем или нижнем регистре), а `{ref}` — либо заголовочное
слово (в любой поддерживаемой транслитерации), либо `lnum`. Запрос разрешается в те же
записи, что и `/dicts/{id}/restful/entries`: HTML для браузеров и JSON-обертка §3.2 при
согласовании содержимого (`Accept: application/json`). Одна схема ссылок служит и людям, и
машинам, поэтому работа делается один раз. Сегменты `restful` и `graphql` зарезервированы и
не являются допустимыми значениями `{dict}`.

## 7. Реестр идентификаторов словарей

Идентификаторы — в нижнем регистре. Семь идентификаторов C-SALT (все проверены как
доступные 11.06.2026) и соответствующие коды CSL:

| id Salt | Код CSL | Направление |
|---|---|---|
| `mw` | MW | санскрит→англ. |
| `ap90` | AP90 | санскрит→англ. |
| `bhs` | BHS | санскрит→англ. |
| `vei` | VEI | санскрит→англ. |
| `pwg` | PWG | санскрит→нем. |
| `gra` | GRA | санскрит→нем. |
| `ae` | AE | англ.→санскрит |

Хост CSL СЛЕДУЕТ дополнительно предоставлять любой другой словарь CDSL v02 по тому же
контракту, используя его код в нижнем регистре как id. Покрытие сверх этих семи — главное
преимущество оригинала перед производной версией.

## 8. Объект записи (Entry)

### 8.1 Поля, совместимые с C-SALT (верхний уровень — ДОЛЖНЫ совпадать с C-SALT)

| Поле (REST / GraphQL) | Тип | Смысл |
|---|---|---|
| `id` / `id` | строка | Стабильный id. **ДОЛЖЕН в точности равняться C-SALT:** `lemma-{headword_slp1}` для уникального заголовка или `lemma-{headword_slp1}-{n}` для омонимов (`n` — номер омонима с 1; проверено: `ka` → `lemma-ka-1`…`-4`). Восстанавливается из CSL `key` + счетчика омонимов `hc1`. |
| `headword_slp1` / `headwordSlp1` | строка | Заголовочное слово в SLP1. |
| `sense` / `sense` | строка[] | Толкования (значения). |
| `re_headwords_slp1` / `reHeadwordsSlp1` | строка[] | Подзаголовки / производные заголовки в SLP1. |
| `created` / `created` | дата-время | Отметка времени индексации/сборки. |
| `xml` / `xml` | строка \| null | **Тело статьи в TEI-P5.** На хосте CSL это ДОЛЖНО быть `null`, пока не выпущена конверсия в TEI (дорожная карта, фаза 5), и НЕ ДОЛЖНО содержать display-XML CSL. |

### 8.2 Расширение CSL (объект `csl` — дополнение; отсутствует в C-SALT)

Хост CSL ДОЛЖЕН помещать все данные, специфичные для оригинала, в единый объект `csl`,
чтобы верхний уровень оставался полностью совместимым. Клиент C-SALT ДОЛЖЕН иметь
возможность игнорировать `csl` без потери данных, определенных C-SALT.

| Поле `csl.` | Из CSL | Смысл |
|---|---|---|
| `lnum` | `lnum` | Кельнский id записи / L-номер (например, `144239`); равен TEI `monier_<lnum>` и параметру `lnum` Funderburk’а. |
| `page` / `column` | `pageNumber` / `columnId` | Координаты скана. |
| `scanUrl` | `imgUrl` | «Чистая» ссылка Salt на отсканированную страницу. |
| `html` / `text` | `html` / `text` | Отрисованная статья. |
| `xmlCsl` | `xml` | Display-XML CSL (кельнская разметка издания) — доступна сразу. |
| `references` | `references[]` | Плоский список меток источников/ссылок. |
| `headwordDeva` / `headwordIast` / `accentedKey` | производные | Транслитерации и форма key2 с ударением. |

## 9. Заявление о совместимости

Запрос, корректный для C-SALT и направленный совместимому хосту CSL, ДОЛЖЕН вернуть ответ,
структура верхнего уровня которого (`data.entries[]` / `data.ids[]`) и поля записи (§8.1)
неотличимы по структуре от ответа C-SALT, за исключением того, что:
1. `xml` МОЖЕТ быть `null` до фазы 5; и
2. каждая запись МОЖЕТ нести дополнительный объект `csl`.

Иные структурные расхождения профилем не допускаются. Поведенческие расхождения (покрытие
дополнительных словарей, дополнительные омонимы, более богатые данные `csl`) ожидаемы и
каталогизированы в [`SALT_API_LOSS_REPORT.md`](SALT_API_LOSS_REPORT.md).

## 10. Версионирование

Профиль версионируется независимо от любого хоста. Обратносовместимые добавления (новые
поля `csl.*`, новые id словарей) увеличивают минорную версию; изменения §8.1 или §2–§6 —
мажорную.
