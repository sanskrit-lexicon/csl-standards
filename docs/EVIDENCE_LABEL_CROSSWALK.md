# Evidence-Label Crosswalk

Date: 2026-06-13

Status: draft. Documentation crosswalk; no exporter ships with it.

This document maps the **epistemic evidence vocabulary** used across `csl-atlas`
— the per-claim evidence labels and the review-report statuses — onto two
external standards: the **W3C PROV Ontology (PROV-O)** and the **TEI** mechanism
for certainty and responsibility (`@cert`, `@evidence`, `@resp`). It makes the
atlas's otherwise private vocabulary citable, interoperable, and reviewable
against published standards (FAIR *Interoperable* + *Reusable*), and addresses
the `csl-atlas` review gap G3 ("the in-house evidence vocabulary is unmapped").

## Trust Block

- Evidence: `derived` — hand-authored from the cited specifications and the
  committed `csl-atlas` vocabulary; no generated data.
- Limitations: a crosswalk, not an RDF/TEI serialization; PROV-O cannot by
  itself grade certainty (see §A); several status mappings are annotations, not
  core PROV relations (flagged inline).
- Validation: Turtle examples use the prefixes already declared in
  [`data/schema/ontolex-frac-profile.shacl.ttl`](../data/schema/ontolex-frac-profile.shacl.ttl);
  not machine-validated here.
- Owner repo: `csl-standards` (standards mapping is out of scope for `csl-atlas`
  per [`docs/BOUNDARY_RULES.md`](BOUNDARY_RULES.md)).
- Next use: a future profile revision can enforce the label→`@cert` and
  label→`prov:` mappings via the archival ODD and the OntoLex/FrAC SHACL shapes.

## 1. Two orthogonal axes

`csl-standards` already defines an **evidence-class** taxonomy in
[`INTEROPERABILITY_MODEL.md`](INTEROPERABILITY_MODEL.md#evidence-classes)
(`textual-source`, `named-kosha`, `generic-lexicographer`, …). That axis answers
*what kind of source* licenses a claim, and is carried in RDF by
`csl:evidenceType` on a `frac:Attestation`.

This crosswalk concerns a **different, orthogonal axis**: *how well established a
value is*, regardless of its source kind. The two compose — a single value may be
`(evidenceType = named-kosha, evidenceLevel = observed)` or
`(evidenceType = generic-lexicographer, evidenceLevel = inferred)`. To keep them
distinct in RDF this document introduces:

- `csl:evidenceLevel` — the epistemic grade (`observed` | `derived` | `inferred` | `reviewed`);
- `csl:reviewStatus` — the human-review lifecycle state (six values, §B).

Both sit on the value node alongside the existing `csl:evidenceType` on its
attestation.

## 2. Source vocabularies (authoritative in `csl-atlas`)

- Evidence labels — `csl-atlas/docs/EVIDENCE_LABELS.md`; enum mirrored in
  `csl-atlas/data/schema/review-report.schema.json` (`evidenceLevel`):
  `observed`, `derived`, `inferred`, `reviewed`.
- Review statuses — `csl-atlas/docs/REVIEW_REPORTS.md`; same schema
  (`reviewStatus`): `machine`, `needs-review`, `reviewed-ok`,
  `reviewed-corrected`, `blocked`, `deferred`.
- Dataset envelope — every generated dataset carries `schemaVersion`,
  `generatedAt`, `sourcePath`, `recordCount`, `assumptions`, `warnings`
  (`csl-atlas/ARCHITECTURE.md`; `scripts/lib/review-report.mjs`).

## A. Evidence labels → PROV-O → TEI

| `evidenceLevel` | Definition (EVIDENCE_LABELS.md) | PROV-O expression | TEI (`@cert` / `@evidence` / `@resp`) |
|---|---|---|---|
| `observed` | Directly present in a dictionary source record | value `prov:wasQuotedFrom` the `csl:SourceRecord`; `prov:hadPrimarySource` that record; the quoting activity performs no transformation | `cert="high"` `evidence="internal"`; `@resp` → the source/transcription |
| `derived` | Produced by a fixed, reproducible rule from observed data | value `prov:wasGeneratedBy` a deterministic `prov:Activity` that `prov:used` the observed entity and `prov:wasDerivedFrom` it; activity `prov:wasAssociatedWith` a `prov:SoftwareAgent` (the build script) | `cert="high"` `evidence="internal"`; `@resp` → the algorithm (`<respStmt><resp>automatic</resp></respStmt>`) |
| `inferred` | A useful heuristic that has not been verified; may be wrong | as `derived`, but the `prov:Derivation` is heuristic; **PROV-O cannot grade it** — the uncertainty lives only in `@cert` / `evidenceLevel` | `cert="low"` (or `"medium"`) `evidence="conjecture"`; `@resp` → the heuristic |
| `reviewed` | A human has confirmed or corrected the value | value `prov:wasAttributedTo` a `prov:Person`; generated by a review `prov:Activity` `prov:wasAssociatedWith` that person; a correction is a `prov:Revision` (`prov:wasRevisionOf` + `prov:wasInvalidatedBy` on the prior value) | `cert="high"` `@resp="#reviewer"`; a correction → `<choice><sic resp="#machine"/><corr resp="#reviewer"/></choice>` |

**Key consequence.** PROV-O separates `observed` / `derived` / `reviewed`
*structurally* — quotation vs. derivation vs. human attribution — but it has **no
native way to separate `derived` from `inferred`**: both are derivations by a
rule. That distinction is precisely the confidence grade carried by TEI `@cert`
and made compact by the atlas `evidenceLevel`. The three vocabularies are
therefore complementary, not redundant: PROV-O records the *generation graph*,
`@cert` records the *confidence*, and `evidenceLevel` is the atlas's one-token
join of both.

## B. Review statuses → PROV-O → TEI

| `reviewStatus` | Meaning (REVIEW_REPORTS.md) | PROV-O expression | TEI |
|---|---|---|---|
| `machine` | Produced by heuristics; no human has looked at it | only `prov:SoftwareAgent` attribution; no review activity exists yet | `@resp="#machine"`; `@cert` inherited from `evidenceLevel` |
| `needs-review` | Flagged for a human (low confidence or conflicting sources) | a review-request annotation entity `prov:wasDerivedFrom` the value; **no `prov:Person` yet** (annotation, not a core relation) | `<certainty cert="unknown"/>` + `<note>needs-review</note>` |
| `reviewed-ok` | A human confirmed the machine value | review `prov:Activity` `prov:used` the value and `prov:wasAssociatedWith` a `prov:Person`; the value is now additionally `prov:wasAttributedTo` that person (endorsement) | `cert="high"` `@resp="#reviewer"` |
| `reviewed-corrected` | A human supplied a corrected value | the corrected value `prov:wasRevisionOf` the machine value; the prior value `prov:wasInvalidatedBy` the review activity | `<choice><sic resp="#machine"/><corr resp="#reviewer"/></choice>` |
| `blocked` | Cannot be resolved yet (bad source data, missing evidence) | review `prov:Activity` ended unresolved; the value's certainty stays unknown (annotation) | `<certainty cert="unknown"/>` + `<note>blocked</note>` |
| `deferred` | Valid but postponed to a later phase | a process/plan note (`prov:Plan`); **no change to the value's provenance** | editorial `<note type="status">deferred</note>` |

The lifecycle relates to the epistemic axis but is not identical to it: a value
is normally `(evidenceLevel ∈ {observed, derived, inferred}, reviewStatus = machine)`
before review, and `(evidenceLevel = reviewed, reviewStatus ∈ {reviewed-ok, reviewed-corrected})`
after it.

## C. Dataset envelope → PROV-O / DCMI / VoID

| Envelope field | Meaning | PROV-O | DCMI / other |
|---|---|---|---|
| `schemaVersion` | shape version of the dataset | — | `dct:conformsTo` (the JSON Schema) |
| `generatedAt` | ISO-8601 generation timestamp | `prov:generatedAtTime` (`xsd:dateTime`) | `dct:created` |
| `generatedBy` / `sourceGeneratedBy` | the build command | `prov:wasGeneratedBy` → `prov:Activity`; activity `prov:wasAssociatedWith` a `prov:SoftwareAgent` | `pav:createdWith` |
| `sourcePath` | source dictionary file | `prov:hadPrimarySource` / `prov:wasDerivedFrom` | `dct:source` |
| `recordCount` | number of records | — | `void:entities` |
| `queue` | review-queue identifier | a `prov:Activity` category | `dct:type` |
| `assumptions[]` | fixed-rule assumptions | annotation on the activity | `skos:scopeNote` |
| `warnings[]` | caveats / known issues | annotation | `skos:note` |
| repository licence | reuse terms | — | `dct:license` = `CC-BY-SA-4.0` |
| owner repo | publisher | `prov:wasAttributedTo` (organisation) | `dct:publisher` |

Prefixes beyond the repo's existing set: `pav: <http://purl.org/pav/>`,
`void: <http://rdfs.org/ns/void#>`, `skos: <http://www.w3.org/2004/02/skos/core#>`,
`xsd: <http://www.w3.org/2001/XMLSchema#>`.

## 3. Worked example

### 3.1 A `derived` value

`csl-atlas` classifies an MW record as gender `m` because it carries
`<lex>m.</lex>` — `evidenceLevel = derived`, `reviewStatus = machine`
(EVIDENCE_LABELS.md's own example).

Atlas JSON (envelope abridged):

```json
{
  "schemaVersion": "1.0.0",
  "generatedAt": "2026-06-13T00:00:00.000Z",
  "sourcePath": "../csl-orig/v02/mw/mw.txt",
  "generatedBy": "npm run build-mw-depth",
  "records": [
    { "recordId": "mw:12345", "gender": "m", "evidenceLevel": "derived" }
  ]
}
```

PROV-O (Turtle):

```turtle
@prefix prov: <http://www.w3.org/ns/prov#> .
@prefix dct:  <http://purl.org/dc/terms/> .
@prefix xsd:  <http://www.w3.org/2001/XMLSchema#> .
@prefix csl:  <https://sanskrit-lexicon.github.io/csl-standards/ns#> .

csl:value_mw_12345_gender
    a prov:Entity ;
    prov:value "m" ;
    csl:evidenceLevel "derived" ;
    csl:reviewStatus  "machine" ;
    prov:wasGeneratedBy csl:act_mwDepthBuild ;
    prov:wasDerivedFrom csl:record_mw_12345 .

csl:act_mwDepthBuild
    a prov:Activity ;
    prov:used csl:record_mw_12345 ;
    prov:wasAssociatedWith csl:agent_mwDepthBuild ;
    prov:generatedAtTime "2026-06-13T00:00:00Z"^^xsd:dateTime .

csl:agent_mwDepthBuild
    a prov:SoftwareAgent ;
    dct:title "npm run build-mw-depth" .

csl:record_mw_12345
    a csl:SourceRecord, prov:Entity ;
    csl:dictionary "mw" ;
    dct:source <https://github.com/sanskrit-lexicon/csl-orig/blob/master/v02/mw/mw.txt#L12345> .
```

TEI (archival fragment):

```xml
<gramGrp>
  <gram type="gen" cert="high" resp="#mw-depth-build">m</gram>
</gramGrp>
<!-- in <teiHeader> -->
<respStmt xml:id="mw-depth-build">
  <resp>automatic classification from &lt;lex&gt;</resp>
  <name>npm run build-mw-depth</name>
</respStmt>
```

### 3.2 A `reviewed` value

The R2 checkpoint decision `r2-drift:gam:pwg` was confirmed by a human reviewer
(`reviewer = gasyoun`, `reviewStatus = reviewed-ok`, so the decided value is
`evidenceLevel = reviewed`).

```turtle
csl:value_r2_gam_pwg
    a prov:Entity ;
    csl:evidenceLevel "reviewed" ;
    csl:reviewStatus  "reviewed-ok" ;
    prov:wasGeneratedBy csl:act_r2CheckpointReview ;
    prov:wasAttributedTo csl:person_gasyoun .

csl:act_r2CheckpointReview
    a prov:Activity ;
    prov:used csl:value_r2_gam_pwg_machine ;
    prov:wasAssociatedWith csl:person_gasyoun ;
    prov:generatedAtTime "2026-06-12T00:00:00Z"^^xsd:dateTime .

csl:person_gasyoun a prov:Person ; dct:title "gasyoun" .
```

TEI: the confirmed value carries `cert="high" resp="#gasyoun"`; had the reviewer
corrected it, the prior machine value would appear inside
`<choice><sic resp="#machine"/><corr resp="#gasyoun"/></choice>`.

## 4. How this attaches to the existing profiles

- **OntoLex/FrAC SHACL** ([`ontolex-frac-profile.shacl.ttl`](../data/schema/ontolex-frac-profile.shacl.ttl))
  already requires `prov:wasDerivedFrom` on every `frac:Attestation` and
  `dct:source` on every `csl:SourceRecord`. `csl:evidenceLevel` and
  `csl:reviewStatus` attach to the value/attestation node as additional
  properties; a future profile revision can add
  `sh:property [ sh:path csl:evidenceLevel ; sh:in ( "observed" "derived" "inferred" "reviewed" ) ]`.
- **TEI archival ODD** ([`tei-archival-profile.odd.xml`](../data/schema/tei-archival-profile.odd.xml))
  governs `@cert` / `@resp`; the label→`@cert` mapping in §A is what a Schematron
  rule in the ODD would enforce (e.g. `evidenceLevel="inferred"` ⇒ `@cert` not
  `"high"`).
- **Neutral model** ([`INTEROPERABILITY_MODEL.md`](INTEROPERABILITY_MODEL.md)):
  `csl:evidenceType` (citation class) and `csl:evidenceLevel` (epistemic grade)
  are kept as separate properties, per §1.

## 5. Caveats and boundary

- This is a documentation crosswalk, not a serializer. No RDF or TEI is
  generated here; a real exporter would extend the ODD / SHACL profiles above.
- PROV-O records *how* a value was produced, not *how confident* we are in it.
  The `derived` vs `inferred` distinction therefore has no core PROV relation and
  must travel in `@cert` / `evidenceLevel`. Status mappings marked "annotation"
  (`needs-review`, `blocked`, `deferred`) likewise have no single core PROV
  relation and are modelled as annotation entities.
- The mappings are deliberately conservative: where a faithful PROV/TEI
  expression would over-claim (e.g. asserting human attribution for a
  `needs-review` row that no person has touched), the crosswalk withholds it.
- Owner repo: `csl-standards`. `csl-atlas` keeps its JSON envelope and labels
  unchanged; this repository owns the standards mapping.
